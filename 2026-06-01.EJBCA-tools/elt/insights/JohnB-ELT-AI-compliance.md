## Purpose of This Document

This document addresses corporate AI compliance requirements for
`ejbca-lifecycle-tool.py` (`elt`), a Python command-line tool developed
with AI pair-programming support. It maps the tool's development and
operational practices to the ten guidelines in the corporate
**PoC for Development LLMs (on Prem)** policy document, and provides
notes for security review.

### Development Context

Primary development was carried out offsite, on the developer's private
infrastructure (Mountain Informatik GmbH), using publicly available
information and authorized code from the software vendor (Keyfactor /
EJBCA). No corporate systems, data, or network resources were involved
in development.

Before any use in the corporate environment, the tool is subjected to:

- Functionality testing against the developer's own EJBCA lab instances
- Static security analysis (Bandit, pip-audit — see section 3)
- Manual review of all changes

Only after this vetting process is the script made available for careful,
supervised use in the corporate DEV/TEST environment, operating exclusively
with EJBCA via its published REST API using a restricted access certificate.

Integration of this development workflow with the corporate AI-LLM PoC
infrastructure is pending.

In the interim, this tool and associated compliance documentation are
delivered as a vetted external contribution —
consistent with the established pattern of open-source collaboration with
the Keyfactor user community.

## 1. Responsibility (Guideline 1)

> *"The responsibility for the AI-generated code or output always remains
> clear with you as a human being."*

All code in `ejbca-lifecycle-tool.py` has been reviewed, tested, and
accepted by John Buehrer (JohnB) as the responsible developer and
system owner. The tool is not deployed autonomously — it is invoked
manually by the responsible administrator.

Read operations are the common case. Write operations (certificate
revocation, End Entity deletion) require an explicit `-commit` flag and
are always presented as a dry-run summary first. This means the
administrator reviews the proposed action before it is executed.
Additionally, the tool can be used with a read-only EJBCA access
certificate, in which case any write request is cleanly rejected by
EJBCA with a clear error message — a safe and expected failure mode.

## 3. SAST — Static Application Security Testing (Guideline 3)

> *"All AI-generated code components must be automatically scanned for
> vulnerabilities, privacy risks, and coding errors."*

### Recommended SAST tool: Bandit

**Bandit** is the standard Python SAST tool, maintained by the
Python Security community.

```bash
pip install bandit
bandit ejbca-lifecycle-tool.py -ll    # medium and high severity only
bandit ejbca-lifecycle-tool.py        # full scan including low severity
```

**Confirmed scan results for v3.12.0** (Bandit 1.9.4, Python 3.13.11,
macOS, 2026-04-01):

| Severity | Count | Notes |
|----------|-------|-------|
| High | 0 | — |
| Medium | 0 | — |
| Low | 2 | See findings table below |

**Low-severity findings — explained and accepted:**

| ID | Finding | Explanation |
|----|---------|-------------|
| B404 | `import subprocess` | Subprocess module imported. Used exclusively for `kubectl` invocation in the optional `-k8s-compare` feature. Arguments are passed as a list (not a shell string), which is the safe pattern. |
| B603 | `subprocess.run()` without `shell=True` | Flagged as a reminder to verify that no untrusted input reaches the command. Confirmed: the `kubectl` command is assembled from validated CLI arguments only. `shell=True` is deliberately absent — this is the secure pattern, not a gap. |

### Additional transparency — items not flagged by Bandit, documented proactively:

| Topic | Detail | Rationale |
|-------|--------|-----------|
| SSL certificate verification | The `-no-verify-ssl` flag disables TLS certificate verification for the EJBCA connection | `elt` is an *inspection* tool; it intentionally connects to servers with self-signed, expired, or otherwise untrusted certificates to inspect them. This is user-controlled and documented in `--help`. |
| File read (`open()`) | One `open()` call reads an optional cert-manager Issuer YAML file | The file path is supplied explicitly by the user via the `-f` flag. Read-only, local file, under the user's control. No path traversal risk: the path is used directly as supplied by the operator. |

### Dependency audit: pip-audit

```bash
pip install pip-audit
pip-audit -r requirements.txt
```

A `requirements.txt` is provided alongside this document listing the
core runtime dependencies (`requests`, `urllib3`). Run pip-audit against
it to verify no known CVEs in the installed dependency versions.

Note: `pip-audit ejbca-lifecycle-tool.py` does not work — pip-audit
expects a project file, not a script. Use `-r requirements.txt` instead.

### Usage without CI/CD pipeline

As this tool is used manually rather than via an automated pipeline,
the recommended practice is to run Bandit before each new version is
put into use:

```bash
bandit ejbca-lifecycle-tool.py -ll -o bandit-report-v3.12.0.txt
pip-audit -r requirements.txt
```

Retain the Bandit report as an audit artefact alongside the version
being deployed.

## 6. Explainability (Guideline 6)

> *"Make sure that every AI decision is understandable and explainable
> to humans ('Explainability-by-Design')."*

`ejbca-lifecycle-tool.py` is a deterministic command-line tool with
no AI inference at runtime. All logic is explicit Python code, readable
and auditable. The tool does not make autonomous decisions — it queries
EJBCA, formats and displays data, and optionally performs revocation
or deletion operations that are always dry-run by default and require
an explicit `-commit` flag to execute.

Output is designed to be self-explanatory: every report includes a
`Generated:` timestamp, `Host:` identifier, and all active filter
parameters in the banner, so any archived output is fully traceable
to its context.

## Security Profile

### Access model

`ejbca-lifecycle-tool.py` connects to the EJBCA REST API using an
mTLS client certificate. The typical configuration uses the same
**RA Administrator** role certificate used by cert-manager customers,
which allows both read operations and write operations (revocation,
End Entity deletion) within the authorised profiles.

For environments where only read access is desired, the tool works
equally well with a restricted read-only certificate. In that case,
any write request (`-commit`) will be cleanly rejected by EJBCA with
an error message — a safe and expected failure mode, not a tool defect.

In all cases, write operations require the explicit `-commit` flag.
Without it, the tool runs in dry-run mode and makes no changes to EJBCA.

### Filesystem access

The tool's own code opens exactly one file: an optional cert-manager
Issuer YAML file, supplied explicitly by the user via the `-f` flag.
This is a read-only operation on a user-specified local file.
Certificate and key files are handled entirely by the `requests`
library via file path references — ELT itself never reads their
contents.

The tool does not write to the filesystem, create temporary files,
or access any system files.

### Network access

All network activity is HTTPS (TLS) to the EJBCA host specified by
the operator. The optional `-k8s-compare` feature additionally invokes
`kubectl` to query the local Kubernetes API server. No other outbound
connections are made. The `-no-proxy` flag bypasses system proxy
settings when needed.

### Dependency surface

Core dependencies: `requests`, `urllib3`. No native extensions, no C
libraries beyond the Python standard library SSL bindings. Run
`pip-audit -r requirements.txt` to verify no known CVEs in installed
dependency versions before deployment.
